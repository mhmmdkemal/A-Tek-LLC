SDLC Handbook

The purpose of the SDLC Process is to help any organization achieve a sounder, better-documented, more secure software development process.

This guide is intended to be utilized by the Project Management Office and Applications Team in an organization, as it sets standards, provides guidance, and establishes approved procedures for software development.

Planning and Initiation Templates

The purpose of the Initiation and Planning phase is to express a software development need and to document the purpose of the needed software. A plan is developed that documents the approach to be used and includes a discussion of methods, tools, tasks, resources, project schedules, and user input. Personnel assignments, costs, project schedule, and target dates are established.

Project planning serves to establish the project scope and objectives and to identify and define the resources required to successfully deliver quality products and services to the users. This activity results in a foundation that will enable project leadership to control cost, schedule, technical performance, and project quality throughout the project lifecycle.

Requirements Template

The intent of the requirements phase is to establish a common understanding between the PO, the PM assigned to the project and the developer tasked with implementing those requirements.

In the Requirements phase the PM meets with stakeholders to elicit their needs, constraints, and requirements and documents them as Business Requirements or User Stories. The PM or Business Analyst (BA) works with the IT Governance Data manager to determine the data required to meet the customer’s requirement and if the data is available. A strategy is developed to capture the data if it is already available or if it needs to be captured from other sources.

Design Templates

The Design phase focuses on defining the target architecture and application level design. The High Level Design defines the architecture components, including the system reference architecture and deployment design. The system design document further elaborates on the architecture as well as the detailed system design including the class diagrams, sequence diagrams, and UI design. Additionally, the system design defines the data architecture including the logical and physical data models and the data dictionary.

Testing Templates

The purpose of the Testing phase is to verify that the delivered product meets the agreed upon requirements. Testing involves both verification and validation of the code developed. Verification is the evaluation of whether or not a product, service, or system complies with a regulation, requirement, specification, or imposed condition and is generally conducted by the development team. Validation is the assurance that a product, service, or system meets the needs of the customer and other identified stakeholders and is often conducted by external users. Programmers verify to show that the delivered product has been built according to the requirements and design specifications; independent testers validate to ensure that the product actually meets the user's needs, and that the specifications were correct in the first place.

Deployment Templates

The purpose of the Deployment phase is to complete deployment preparation and Production deployment of the software. The Deployment Description Document, which defines the deployment requirements for the system, is completed. Release Notes, which communicate the major new features or changes in the release, are developed. The Production Verification Plan, which documents the steps to verify the completion and success of a software deployment, is generated, as well as the Production Back Out Plan, which documents how the team can reverse the migration, if necessary. A User Guide, which describes how to use the software/system from an end user perspective, is developed.
